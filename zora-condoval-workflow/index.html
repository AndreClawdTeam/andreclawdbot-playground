Visão Geral e Objetivo

O workflow R2 — Conditional Approval Processing automatiza um dos processos mais críticos e trabalhosos da plataforma HouseNumbers: receber um email de conditional approval do lender, extrair as condições exigidas, criar as tasks correspondentes no sistema, cruzar com documentos já disponíveis no loan e draftar emails de outreach para coletar os docs faltantes — tudo isso sem intervenção manual do MLP.

💡
Problema que resolve

Hoje um MLP recebe o email de condval, lê manualmente cada condição, entra no sistema pra criar task por task, verifica quais docs já existem no loan, identifica quem contatar pra cada condição e digita os emails de outreach um a um. São horas de trabalho mecânico. O R2 faz tudo isso automaticamente.

FLUXO END-TO-END (OPÇÃO B — EVENT-DRIVEN)

📧 Email de Conditional Approval
   └── chega no Front (inbox do loan) com PDF em anexo

🔗 Front Webhook
   └── POST → communication-service
          └── publica WEBHOOK_SYNC_FRONT_CONVERSATION no SNS

⚡ agent-service (consumer do SQS)
   └── recebe o evento
          └── registry lookup → encontra R2

⚡ Tier 1 Check (sem LLM, O(1), determinístico)
   └── O email tem um PDF em anexo?
          ├── NÃO → SKIP (sem LLM, sem custo)
          └── SIM → executa Tier 2

🤖 Tier 2 — Mastra Agent (R2)  maxSteps=15
   ├── 1. get_conversation(conversationId)   ← busca conversa + PDF
   ├── 2. LLM classifica: "É um condval?"   ← structured output
   │       └── isConditionalApproval===false → retorna []
   ├── 3. prepare_condoval_import            ← get existing tasks + dedup
   ├── 4. Extrai N condições do PDF
   ├── 5. list_tasks                         ← checa duplicatas
   ├── 6. create_task × N                    ← loop: uma call por condição
   ├── 7. list_documents                     ← docs já upados no loan
   ├── 8. LLM raciocina: doc satisfaz task?  ← update_task(linkedDocuments)
   ├── 9. suggest_condition_destinations     ← LLM + structured output
   ├── 10. draft_condition_outreach × K      ← drafta email por condição
   └── 11. create_comment                    ← nota interna com resumo

📤 R2 dispara TASK_CREATED × N

🤖 Tier 2 — Mastra Agent (R7)  maxSteps=10
   ├── TASK_CREATED trigger → doc matching refinado
   ├── update_task(linkedDocuments) para matches
   └── DRAFT_EMAIL para tasks sem doc resolvido

📤 ProposedActions retornadas ao pipeline:
   ├── CREATE_TASK × N           → autoExecute (threshold=0)
   ├── DRAFT_EMAIL × K           → autoExecute (threshold=0)
   └── CREATE_COMMENT            → autoExecute (threshold=0)

🔍 MLP Review
   └── inbox mostra tudo com confidence scores
          └── MLP aprova, edita ou descarta

🏗️ Onde vive

agent-service — long-running ECS Service. Core do sistema de agentic workflows. Padrão handler puro: recebe PipelineContext, retorna ProposedAction[].

WEBHOOK_SYNC_FRONT_CONVERSATION

Único trigger. O Tier 1 filtra se há PDF em anexo.

Como Integrar com o AI Agent

📡 Trigger: apenas WEBHOOK_SYNC_FRONT_CONVERSATION

✅
Trigger único e correto

O R2 tem apenas WEBHOOK_SYNC_FRONT_CONVERSATION como trigger. Quando um email chega no Front, esse evento é disparado. O Tier 1 filtra os irrelevantes (sem PDF = não é condval). O LOAN_APPLICATION_UPSERTED foi removido completamente — era um fallback para import manual que não faz sentido aqui.

🔍
Investigação: tools de batch e doc-task match não existem no MCP

André perguntou: "Não temos uma tool de batch creation e uma de document-task match?" — A resposta é não. Veja como o workflow resolve isso:

  • Batch task creation: não existe. Só create_task (singular). O Mastra Agent cria tasks em loop — uma call por condição — usando prepare_condoval_import antes para deduplicação. Esta é a abordagem correta: o agent gerencia a iteração.
  • Document-task matching: não existe como tool separada. O R7 implementa o matching usando list_documents + list_tasks + raciocínio LLM → update_task com linkedDocuments. Não há uma tool de "match document to task" — o LLM raciocina sobre o match.

Antes vs. Depois

🔴 Antes (Manual)

  • MLP recebe email de condval
  • Lê cada condição manualmente
  • Cria tasks uma a uma no sistema
  • Verifica docs disponíveis manualmente
  • Digita emails de outreach individualmente
  • ~2-4 horas por conditional approval

🟢 Depois (R2 Automatizado)

  • Email com PDF chega → Tier 1 passa
  • LLM classifica PDF (structured output)
  • Agent extrai todas as condições
  • Tasks criadas em loop automaticamente
  • Docs cross-matched via raciocínio LLM
  • MLP revisa e aprova: ~10-15 min

⚡ Tier 1: único check determinístico

💡
Por que só checar PDF?

Conditional approval emails sempre vêm com o PDF da conditional approval em anexo. Se não tem PDF, não é um condval — ponto. Esta checagem é O(1), sem LLM, sem ambiguidade. A classificação do tipo do email acontece dentro do Tier 2 (primeira ação do Mastra Agent), usando o PDF recuperado com structured output.

Tier 1 Check — O(1), sem LLM, determinístico

Único check: email-has-pdf-attachment
   → payload.attachments.some(a => a.contentType === "application/pdf")

   → false: SKIP (não é condval, sem gastar tokens)
   → true: executa Tier 2

Tier 2 — PRIMEIRA ação do Mastra Agent (classificação LLM)

   get_conversation(conversationId) → busca conversa com metadata do PDF
   LLM classifica via structured output:
   {
     isConditionalApproval: boolean,
     confidence: number,
     signals: string[]   // evidências encontradas no PDF
   }

   → isConditionalApproval===false: handler retorna [] (early return)
   → true: continua extração de condições

Arquitetura: Opção A vs. Opção B

Opção A Monolítica

R2 faz tudo em um único handler: extração → tasks → doc matching → email drafting. Um único Mastra Agent com maxSteps=15+.

  • ✅ Mais simples de implementar
  • ✅ Um único ponto de controle
  • ✅ Context compartilhado no agente
  • ❌ Handler grande e difícil de testar
  • ❌ Não reutiliza R7 para outros cases
  • ❌ Timeout risk com muitas condições
Opção B ⭐ Event-Driven

R2 cria as tasks → TASK_CREATED dispara R7 → R7 faz doc matching e email drafting. Pipeline distribuído.

  • ✅ Alinha com arquitetura existente
  • ✅ R7 reutilizável para outros triggers
  • ✅ Handlers menores e mais testáveis
  • ✅ Resiliente: falha em R7 não afeta R2
  • ❌ Mais complexo de orquestrar
  • ❌ Latência adicional (dois eventos)
✅
Decisão: Opção B (Event-Driven) — confirmada

R2: WEBHOOK_SYNC_FRONT_CONVERSATION → detecta condval → importa condições → CREATE_TASK × N → dispara TASK_CREATED
R7: TASK_CREATED → doc matching → UPDATE_TASK_REVIEW (linked docs) → DRAFT_EMAIL para unresolved

A Opção B alinha melhor com o padrão arquitetural do agent-service e permite que o R7 seja reaproveitado quando qualquer documento chegar — não apenas após um condval. Handlers menores = mais fáceis de testar e manter.

Novas MCP Tools a Criar

2 novas tools precisam ser criadas no mcp-gateway-service. A classificação de email (classify_email_type) foi removida: a classificação agora acontece dentro do handler R2 (Tier 2), como primeira ação do Mastra Agent usando structured output — sem round-trip extra ao MCP gateway.

🎯 Tool 1 — suggest_condition_destinations Query tasks:read + entities:read

Para cada condição, sugere destinatários com base no topic, no texto verbatim da condição e nas entidades vinculadas ao loan. Usa LLM com Structured Outputs — as regras de "quem contatar" são contextuais e o LLM raciocina melhor que regras fixas.

💡
Por que LLM e não rule-based?

Uma condição de "Income" pode requerer contato com o borrower, o employer, ou o CPA dependendo do texto exato. Exemplo: "Provide 2 years W-2s" → borrower. "Verify self-employment income" → borrower + CPA accountant. Regras fixas não conseguem capturar essa nuança — o LLM raciocina sobre o texto verbatim.

Input schema TypeScript 
// Input
{
  applicationId: string,
  conditions: Array<{
    taskId: string,
    conditionText: string,
    timingCode: "PTD" | "PTF" | "PTC" | "UW",
    topic: "Income" | "Assets" | "Credit" | "Property"
         | "Title" | "Insurance" | "Legal" | "Compliance" | "Other"
  }>,
  loanContext: {
    borrowerName: string,
    propertyAddress: string,
    titleCompany?: { name: string, email: string },
    lender?: { name: string, email: string },
    loanOriginator?: { name: string, email: string }
  }
}
Output schema (Structured Output) TypeScript 
// Enum de tipos de destinatário — permite agrupamento
type RecipientType =
  | "borrower"
  | "employer"
  | "title_company"
  | "lender"
  | "insurance_provider"
  | "appraiser"
  | "cpa_accountant"
  | "attorney"
  | "underwriter";

type ConditionDestination = {
  taskId: string;
  suggestions: Array<{
    recipientType: RecipientType;   // enum — permite agrupamento
    recipientName: string;
    recipientEmail?: string;
    confidence: number;             // 0-1
    rationale: string;
  }>;
};

type SuggestDestinationsOutput = {
  destinations: ConditionDestination[];
};
Implementação interna da tool TypeScript 
// suggest-condition-destinations.ts
// Internamente a tool:
// 1. Busca dados do loan via API (title company, lender, borrower, MLO)
// 2. Monta prompt com contexto (condições + entidades disponíveis)
// 3. Chama LLM com structured output (JSON schema enforcement)
// 4. Retorna lista estruturada de destinatários por condição

async function suggestConditionDestinations(
  input: SuggestDestinationsInput
): Promise<SuggestDestinationsOutput> {
  // 1. Buscar entidades do loan
  const loanData = await getLoan(input.applicationId);

  // 2. Montar prompt
  const prompt = buildDestinationPrompt(input.conditions, loanData);

  // 3. LLM com structured output (JSON schema enforced)
  const result = await llm.generateObject({
    schema: destinationsOutputSchema, // zod schema
    prompt,
  });

  return result.object;
}

# Zora — Conditional Approval Processor You are processing a conditional approval email for a mortgage loan. Your job is to: 1. Classify the PDF attachment: is this a conditional approval? (structured output) 2. If yes, extract all conditions from the PDF 3. Check which conditions are already satisfied by uploaded documents 4. For unsatisfied conditions, suggest recipients and draft outreach emails 5. Return a confidence score for each action ## First Step: PDF Classification ALWAYS start by calling get_conversation to retrieve the PDF metadata. Then classify using structured output: isConditionalApproval: boolean confidence: number signals: string[] If isConditionalApproval === false → return [] immediately. Do not proceed. ## Key Rules - ALWAYS call prepare_condoval_import (after classification) for dedup guide - NEVER create duplicate tasks — the dedup guide tells you how to match - conditionDetails MUST be populated on every CREATE_TASK action - create_task is SINGULAR — call it once per condition in a loop - Confidence scoring: - 1.0: exact match / clear action - 0.8-0.99: high confidence, proceed - 0.6-0.79: uncertain, flag for review - <0.6: skip, don't propose action ## Document Matching Use list_documents + list_tasks to reason about which docs satisfy which conditions. There is no dedicated "match" tool — you reason over the lists yourself. Link matched documents via linkedDocuments in the task creation. ## Destination Suggestion For unmatched conditions, call suggest_condition_destinations (uses LLM internally). Draft one outreach email per destination × condition pair using draft_condition_outreach. Always include the condition text verbatim in the draft. ## Final Step Always call create_comment at the end with a summary: - Conditions extracted, tasks created vs. existed, docs matched, outreach drafts created

# Zora — Document-Task Matcher You are matching newly received documents to existing open condition tasks. When a document arrives, check if it satisfies any open tasks for this loan. If a match is found, update the task status and link the document. ## How to Match There is no dedicated matching tool. You reason over the data yourself: 1. Call list_documents to get all documents for this loan 2. Call list_tasks to get all open condition tasks 3. Reason: does document X semantically satisfy condition Y? 4. Call update_task with linkedDocuments to link a match ## Confidence Scoring for Matches - 1.0: Document type exactly matches the condition requirement - 0.85+: Strong semantic match (content clearly addresses the condition) - 0.7-0.84: Partial match — flag for MLP review - <0.7: Don't auto-complete, don't propose action ## Rules - Check ALL open condition tasks, not just the most recent - One document can satisfy multiple conditions - Prefer UPDATE_TASK_REVIEW over UPDATE_TASK_COMPLETE when uncertain - Always include reasoning in your response

r2.ts — Handler principal do R2 TypeScript 
import { WorkflowHandler, PipelineContext } from "../types";
import { buildSoulPrompt, OUTPUT_INSTRUCTIONS } from "../../domain/prompts";
import { pickTools, toMastraTools, toMastraModel } from "../utils";
import { parseProposedActions } from "../parsers";
import { getOrCreateMemory } from "../../memory";
import { Agent } from "@mastra/core";

const R2_TOOLS = [
  // Read — email data + conversation (inclui PDF metadata)
  "list_loan_emails", "get_email", "get_conversation",
  // Read — task dedup guide (MUST call after PDF classification)
  "prepare_condoval_import",
  // Read — existing tasks + docs (para raciocínio de matching)
  "list_tasks", "list_documents", "get_document",
  // Read — loan data + conditions
  "get_loan", "get_loan_conditions",
  // Write — tasks + comments (create_task é SINGULAR, agent faz loop)
  "create_task", "update_task", "create_comment",
  // 🆕 2 novas tools
  "suggest_condition_destinations", "draft_condition_outreach",
] as const;

export const r2Handler: WorkflowHandler = {
  async execute(context: PipelineContext) {
    const instructions = buildSoulPrompt("r2") + OUTPUT_INSTRUCTIONS;
    const mcpToolDefs = await context.getMcpTools();
    const selectedMcp = pickTools(mcpToolDefs, R2_TOOLS);
    const tools = toMastraTools([...context.tools, ...selectedMcp]);

    const agent = new Agent({
      id: "r2-condoval-processor",
      instructions,
      tools,
      model: toMastraModel(context.config.model),
      memory: getOrCreateMemory(),
    });

    const conversationId = context.event.payload.conversationId as string;
    const prompt = `Process the conditional approval for loan: ${context.loanId}
Conversation ID: ${conversationId}

Start by calling get_conversation to retrieve the PDF.
Then classify the PDF (structured output: isConditionalApproval).
If not a condval → return [].
If condval → call prepare_condoval_import, extract conditions, create tasks, match docs, draft outreach.`;

    const result = await agent.generate(prompt, { maxSteps: 15 });
    return parseProposedActions(result.text);
  }
};

import { Tier1CheckDef, PipelineContext } from "../tier1";

export const r2Tier1Checks: Tier1CheckDef[] = [
  {
    name: "email-has-pdf-attachment",
    description: "Conditional approval emails always come with a PDF. No PDF = not a condval.",
    check: async (context: PipelineContext): Promise<boolean> => {
      const payload = context.event.payload;
      // O payload do WEBHOOK_SYNC_FRONT_CONVERSATION inclui metadata da conversa
      // incluindo lista de attachments
      const attachments = (payload.attachments as Array<{ contentType: string }>) ?? [];
      const hasPdf = attachments.some(a => a.contentType === "application/pdf");
      return hasPdf;  // true → Tier 2 | false → SKIP
    }
  }
];
