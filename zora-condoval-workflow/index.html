Visão Geral e Objetivo

O workflow R2 — Conditional Approval Processing automatiza um dos processos mais críticos e trabalhosos da plataforma HouseNumbers: receber um email de conditional approval do lender, extrair as condições exigidas, criar as tasks correspondentes no sistema, cruzar com documentos já disponíveis no loan e draftar emails de outreach para coletar os docs faltantes — tudo isso sem intervenção manual do MLP.

💡
Problema que resolve

Hoje um MLP recebe o email de condval, lê manualmente cada condição, entra no sistema pra criar task por task, verifica quais docs já existem no loan, identifica quem contatar pra cada condição e digita os emails de outreach um a um. São horas de trabalho mecânico. O R2 faz tudo isso automaticamente.

FLUXO END-TO-END

📧 Email de Conditional Approval
   └── chega no Front (inbox do loan)

🔗 Front Webhook
   └── POST → communication-service
          └── publica WEBHOOK_SYNC_FRONT_CONVERSATION no SNS

⚡ agent-service (consumer do SQS)
   └── recebe o evento
          └── registry lookup → encontra R2

⚡ Tier 1 Checks (sem LLM, determinístico)
   ├── ✅ é um conditional approval email? (heurística no subject/metadata)
   └── ✅ não foi processado nos últimos 2h?
          └── ambos OK → executa Tier 2

🤖 Tier 2 — Mastra Agent (R2)  maxSteps=15
   ├── 1. prepare_condoval_import          ← get existing tasks + dedup guide
   ├── 2. get_conversation / get_email     ← lê o email completo
   ├── 3. Extrai N condições do email
   ├── 4. list_tasks                       ← checa duplicatas
   ├── 5. create_task × N                  ← cria cada condição como task
   ├── 6. list_documents                   ← docs já upados no loan
   ├── 7. Cross-match docs × tasks         ← doc satisfaz a condição?
   ├── 8. suggest_condition_destinations × K  ← quem contatar por condição?
   ├── 9. draft_condition_outreach × K     ← drafta email por condição
   └── 10. create_comment                  ← nota interna com resumo

📤 ProposedActions retornadas ao pipeline:
   ├── CREATE_TASK × N           → autoExecute (threshold=0)
   ├── DRAFT_EMAIL × K           → autoExecute (threshold=0)
   └── CREATE_COMMENT            → autoExecute (threshold=0)

🔍 MLP Review
   └── inbox mostra tudo com confidence scores
          └── MLP aprova, edita ou descarta

🏗️ Onde vive

agent-service — long-running ECS Service. Core do sistema de agentic workflows. Padrão handler puro: recebe PipelineContext, retorna ProposedAction[].

📡 Trigger(s)

WEBHOOK_SYNC_FRONT_CONVERSATION
LOAN_APPLICATION_UPSERTED (fallback)

🤖 Model Tier

Powerful GPT-4o ou Claude Sonnet — necessário para extração precisa de condições legais/financeiras do email.

📊 Output esperado

Para um email típico com 8 condições: ~8 CREATE_TASK + ~4 DRAFT_EMAIL (condições sem doc) + 1 CREATE_COMMENT.

2

Como Integrar com o AI Agent

🚨 O Problema do Trigger Atual

O R2 hoje (como stub) tem apenas LOAN_APPLICATION_UPSERTED como trigger. Este evento é disparado por qualquer atualização do loan — mudança de status, edição de campo, qualquer coisa. Usar só ele seria catastrófico: o R2 rodaria dezenas de vezes por dia sem necessidade.

⚠️
Solução: dois triggers + Tier 1 forte

Adicionar WEBHOOK_SYNC_FRONT_CONVERSATION como trigger primário. O Tier 1 filtra se o evento realmente é um conditional approval email. LOAN_APPLICATION_UPSERTED fica como fallback para imports manuais.

Antes vs. Depois

🔴 Antes (Manual)

  • MLP recebe email de condval
  • Lê cada condição manualmente
  • Cria tasks uma a uma no sistema
  • Verifica docs disponíveis manualmente
  • Digita emails de outreach individualmente
  • ~2-4 horas por conditional approval

🟢 Depois (R2 Automatizado)

  • Email chega → webhook dispara R2
  • Agent extrai todas as condições
  • Tasks criadas automaticamente
  • Docs cross-matched em segundos
  • Emails de outreach draftados
  • MLP revisa e aprova: ~10-15 min

⚡ Como o Tier 1 funciona

Tier 1 Checks — rápidos, sem LLM, sem custo de token

Check 1: has-condoval-email-signal
   Para WEBHOOK_SYNC_FRONT_CONVERSATION:
     → subject contém "conditional approval" / "condval" / "approval letter"?
     → sender domain é de um lender conhecido?
     → conversation tags indicam condval?
   Para LOAN_APPLICATION_UPSERTED:
     → loan.status mudou para "conditional_approval"?
     → loan.newConditionsCount > 0?
   
   → false: SKIP (sem gastar tokens)
   → true: continua

Check 2: not-processed-recently
   → context.recentActions tem CREATE_TASK de R2 nas últimas 2h?
   
   → sim: SKIP (idempotência)
   → não: continua → executa Tier 2

Arquitetura: Opção A vs. Opção B

Opção A Monolítica

R2 faz tudo em um único handler: extração → tasks → doc matching → email drafting. Um único Mastra Agent com maxSteps=15+.

  • ✅ Mais simples de implementar
  • ✅ Um único ponto de controle
  • ✅ Context compartilhado no agente
  • ❌ Handler grande e difícil de testar
  • ❌ Não reutiliza R7 para outros cases
  • ❌ Timeout risk com muitas condições
Opção B ⭐ Event-Driven

R2 só cria as tasks → TASK_CREATED dispara R7 → R7 faz doc matching e email drafting. Pipeline distribuído.

  • ✅ Alinha com arquitetura existente
  • ✅ R7 reutilizável para outros triggers
  • ✅ Handlers menores e mais testáveis
  • ✅ Resiliente: falha em R7 não afeta R2
  • ❌ Mais complexo de orquestrar
  • ❌ Latência adicional (dois eventos)
✅
Recomendação: Opção B (Event-Driven)

A Opção B alinha melhor com o padrão arquitetural do agent-service e permite que o R7 (Document Task Auto-Match) seja reaproveitado quando qualquer documento chegar — não apenas após um condval. Handlers menores = mais fáceis de testar e manter.

⚠️
draft_email e send_email NÃO servem para este workflow

draft_email gerencia uma única draft de title order email por loan. O R2 precisa de N drafts independentes, um por condição/destinatário. Por isso precisamos criar draft_condition_outreach como nova tool.

Novas MCP Tools a Criar

Três novas tools precisam ser criadas no mcp-gateway-service.

🎯 Tool 1 — suggest_condition_destinations Query tasks:read + entities:read

Para cada condição, sugere 2-3 destinatários prováveis com base no topic da condição e nas entidades vinculadas ao loan. Sem LLM — lógica rule-based.

Input / Output schema TypeScript
// Input { applicationId: string, conditionTopic: "Income" | "Assets" | "Credit" | "Property" | "Title" | "Insurance" | "Legal" | "Compliance" | "Other", timingCode: "PTD" | "PTF" | "PTC" | "UW", conditionText: string // verbatim condition text } // Output { suggestions: [{ recipient: { name: string, email: string, role: "borrower" | "employer" | "title-company" | "lender" | "insurance-provider" | "appraiser" }, confidence: number, // 0-1 rationale: string }] }

📋 Topic → Destinatários

  • Title → title company
  • Insurance → insurance provider, borrower
  • Income / Assets / Credit → borrower, employer
  • Property → appraiser, title company
  • Legal / Compliance → title company, lender

🔍 Tool 3 — classify_email_type Query emails:read

Classifica o tipo de um email — se é um conditional approval, email geral ou outro tipo. Pode ser usado no Tier 1 para evitar gastos de tokens desnecessários.

Input / Output schema TypeScript
// Input { applicationId: string, conversationId: string } // Output { type: "conditional_approval" | "general" | "other", confidence: number, signals: string[] // ex: ["subject contains 'conditional approval'", ...] }
💡
Alternativa mais simples

A classificação pode ser feita diretamente no Tier 1 check do agent-service (regex/heurística no payload do evento), sem criar esta tool. Isso evita uma nova dependência e latência de rede. Criar a tool só se precisar de classificação mais sofisticada em outros workflows também.

Prompts

📄 domain/prompts/workflows/r2.md

r2.md — Soul Prompt do R2 Markdown
# Zora — Conditional Approval Processor You are processing a conditional approval email for a mortgage loan. Your job is to: 1. Identify and extract all conditions from the email 2. Check which conditions are already satisfied by uploaded documents 3. For unsatisfied conditions, suggest who to contact and draft outreach emails 4. Return a confidence score for each action ## Key Rules - ALWAYS call `prepare_condoval_import` first to get existing tasks and dedup instructions - NEVER create duplicate tasks — the dedup guide tells you exactly how to match - `conditionDetails` MUST be populated on every CREATE_TASK action - Confidence scoring: - 1.0: exact match / clear action - 0.8-0.99: high confidence, proceed - 0.6-0.79: uncertain, flag for review - <0.6: skip, don't propose action ## Document Matching For each new task/condition, call `list_documents` to find documents that might satisfy it. A document satisfies a condition if its type and content plausibly address the condition text. Link matched documents via `linkedDocuments` in the task creation. Set confidence based on how clearly the document addresses the condition. ## Destination Suggestion For unmatched conditions, call `suggest_condition_destinations`. Draft one outreach email per destination × condition pair using `draft_condition_outreach`. Always include the condition text verbatim in the draft. ## Final Step Always call `create_comment` at the end with a summary of: - How many conditions were extracted - How many tasks were created vs. already existed - How many docs matched - How many outreach drafts were created

r7.md — Soul Prompt do R7 (Document Task Auto-Match) Markdown
# Zora — Document-Task Matcher You are matching newly received documents to existing open condition tasks. When a document arrives, check if it satisfies any open tasks for this loan. If a match is found, update the task status and link the document. ## Confidence Scoring for Matches - 1.0: Document type exactly matches the condition requirement - 0.85+: Strong semantic match (content clearly addresses the condition) - 0.7-0.84: Partial match — flag for MLP review - <0.7: Don't auto-complete, don't propose action ## Rules - Check ALL open condition tasks, not just the most recent - One document can satisfy multiple conditions - Prefer UPDATE_TASK_REVIEW over UPDATE_TASK_COMPLETE when uncertain - Always include reasoning in your response

r2.ts — Handler principal do R2 TypeScript
import { WorkflowHandler, PipelineContext } from "../types"; import { buildSoulPrompt, OUTPUT_INSTRUCTIONS } from "../../domain/prompts"; import { pickTools, toMastraTools, toMastraModel } from "../utils"; import { parseProposedActions } from "../parsers"; import { getOrCreateMemory } from "../../memory"; import { Agent } from "@mastra/core"; const R2_TOOLS = [ // Read — email data "list_loan_emails", "get_email", "get_conversation", // Read — task dedup guide (MUST call first) "prepare_condoval_import", // Read — existing tasks + docs "list_tasks", "list_documents", "get_document", // Read — loan data + conditions "get_loan", "get_loan_conditions", // Write — tasks + comments "create_task", "create_comment", // 🆕 New tools "suggest_condition_destinations", "draft_condition_outreach", ] as const; function buildR2Prompt(loanId: string, emailId?: string): string { return ` Process the conditional approval for loan: ${loanId} ${emailId ? `Focus on conversation/email: ${emailId}` : 'Find the most recent conditional approval email.'} Start by calling prepare_condoval_import to understand existing tasks. Then extract all conditions, create tasks, match docs, and draft outreach emails. `; } export const r2Handler: WorkflowHandler = { async execute(context: PipelineContext) { const instructions = buildSoulPrompt("r2") + OUTPUT_INSTRUCTIONS; const mcpToolDefs = await context.getMcpTools(); const selectedMcp = pickTools(mcpToolDefs, R2_TOOLS); const tools = toMastraTools([...context.tools, ...selectedMcp]); const agent = new Agent({ id: "r2-condoval-processor", instructions, tools, model: toMastraModel(context.config.model), memory: getOrCreateMemory(), }); const emailId = context.event.payload.emailId as string | undefined; const prompt = buildR2Prompt(context.loanId, emailId); const result = await agent.generate(prompt, { maxSteps: 15 }); return parseProposedActions(result.text); } };

Gaps e Dependências

O que precisa ser construído antes ou durante a implementação do R2.

🔴

Infraestrutura de condition outreach drafts Blocker

A coleção MongoDB condition_outreach_drafts não existe ainda. Precisa de schema, modelo Mongoose/TypeBox, e CRUD básico antes de implementar a tool draft_condition_outreach. Considerar se vai no task-service ou em um novo microserviço de drafts.

🟠

Novas MCP tools (3) Alto esforço

suggest_condition_destinations, draft_condition_outreach e opcionalmente classify_email_type. Cada uma precisa de implementação, registro no index.ts e testes.

🟠

Novos prompts (r2.md, r7.md) Médio esforço

Os arquivos de soul prompt precisam ser criados em domain/prompts/workflows/. Precisam de iteração com o time para ajustar tom, regras de extração e output format.

🟡

R7 — Document Task Auto-Match (se Opção B) Opcional agora

Se escolher a Opção B (event-driven), o R7 também precisa ser implementado além do R2. O R7 recebe TASK_CREATED e faz o doc matching + email drafting. Se escolher Opção A, R7 fica pra depois.

🟡

Mapeamento de entidades por loan Revisão necessária

A suggest_condition_destinations depende de entidades vinculadas ao loan (title company, lender, loan originator). Verificar se esses vínculos já são acessíveis via get_loan ou se precisam de queries adicionais às entity tools.

🔵

Scopes de autorização Verificar

Verificar se os scopes emails:write para draft_condition_outreach e tasks:read para suggest_condition_destinations já existem no src/tools/index.ts do mcp-gateway-service ou precisam ser adicionados.

🔵

Teste com condval email real Validação

Antes do deploy, testar o handler R2 com um email de conditional approval real (anonimizado) para validar que a extração de condições funciona corretamente com o modelo "powerful". Ajustar prompts conforme necessário.
